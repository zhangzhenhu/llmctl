### 3.2 各模块详细设计

#### 3.2.1 CLI 模块 (`cli/`)

**职责**：命令行参数解析、命令路由、帮助信息展示

**核心逻辑**：

| 组件 | 职责 | 关键点 |
|------|------|--------|
| **Args Parser** | 解析命令行参数 | 使用 clap derive 宏；`--message` 支持多次传入；`-t` 接收1-2个路径参数 |
| **Command Router** | 根据参数路由到对应处理器 | 识别 4 种命令：Call / List / Init / Convert |
| **Help Generator** | 生成帮助信息 | 自动生成，包含参数说明与示例 |

**参数优先级实现逻辑**：

```
优先级判断顺序：
1. 检查命令行参数是否设置 → 最高优先级
2. 检查环境变量是否设置   → 中等优先级
3. 检查配置文件是否设置   → 较低优先级
4. 使用默认值             → 最低优先级

特殊处理：
- --message 参数：不覆盖，而是追加到配置文件 context 末尾
- --stream 参数：布尔标志，存在即为 true
```

**命令识别逻辑**：

| 参数组合 | 命令类型 | 说明 |
|----------|----------|------|
| `-i` 或 `--init` | Init | 初始化配置文件 |
| `-t` 或 `--convert` | Convert | 格式转换 |
| `-l` 或 `--list` | List | 列出模型 |
| 其他情况 | Call | 调用 LLM |

**`-t/--convert` 参数解析逻辑**：

```
参数形式：
├── llm-probe -t input.json          → 输出路径自动推断
└── llm-probe -t input.json out.yaml → 使用指定输出路径

自动推断输出路径规则：
├── 输入: config.json → 输出: config.yaml
├── 输入: config.yaml → 输出: config.json
└── 输入: config.yml  → 输出: config.json

实现方式：
├── 使用 clap 的 value_parser 接收 Vec<PathBuf>
├── 验证参数数量在 1-2 之间
└── 根据参数数量决定输出路径
```

#### 3.2.2 配置模块 (`config/`)

**职责**：配置文件加载、合并、验证、转换

**核心数据结构**：

| 结构 | 用途 | 说明 |
|------|------|------|
| **FileConfig** | 配置文件结构 | 与 YAML/JSON 一一对应，所有字段可选 |
| **RuntimeConfig** | 运行时配置 | 合并后的最终配置，所有字段必填 |
| **Message** | 对话消息 | role + content |

**配置字段规范**：

| 字段 | 类型 | 必填 | 默认值 | 说明 |
|------|------|------|--------|------|
| `provider` | String | 是 | - | 服务商：openai / gemini |
| `base_url` | String | 是 | - | API 基础地址 |
| `api_key` | String | 是 | - | API 密钥（建议用环境变量） |
| `model` | String | 是 | - | 模型名称 |
| `stream` | Boolean | 否 | false | 是否流式返回 |
| `context` | Array | 否 | [] | 对话上下文 |

**Context 结构**：

| 字段 | 类型 | 说明 |
|------|------|------|
| `role` | String | 角色：system / user / assistant |
| `content` | String | 消息内容 |

**配置合并流程**：

```
┌─────────────────────────────────────────────────────────────┐
│                    配置合并流程                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Step 1: 加载默认值                                         │
│  ├── provider: 无                                           │
│  ├── model: ""                                              │
│  ├── stream: false                                          │
│  └── context: []                                            │
│                                                             │
│  Step 2: 加载配置文件（如果存在）                            │
│  ├── 识别格式：YAML / JSON                                   │
│  ├── 解析文件内容                                           │
│  └── 合并到当前配置（覆盖默认值）                            │
│                                                             │
│  Step 3: 读取环境变量                                        │
│  ├── LLM_API_KEY → api_key                                  │
│  ├── LLM_PROVIDER → provider (可选扩展)                     │
│  ├── LLM_BASE_URL → base_url (可选扩展)                     │
│  └── LLM_MODEL → model (可选扩展)                           │
│                                                             │
│  Step 4: 应用命令行参数                                      │
│  ├── 覆盖所有可配置字段                                      │
│  └── --message 追加到 context 末尾（非覆盖）                │
│                                                             │
│  Step 5: 验证必填字段                                        │
│  ├── provider ✓ 必填                                        │
│  ├── base_url ✓ 必填                                        │
│  ├── api_key ✓ 必填                                         │
│  └── model ✓ 必填                                           │
│                                                             │
│  Step 6: 返回 RuntimeConfig                                 │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**格式转换逻辑**：

```
输入文件 → 自动识别格式（扩展名判断）
         ↓
    解析为 FileConfig
         ↓
    ┌────────────────┐
    │ 输出格式判断    │ ← 根据输出文件扩展名
    │ (json/yaml)    │   或默认相反格式
    └────────┬───────┘
             │
    ┌────────┴────────┐
    ▼                 ▼
序列化为 JSON    序列化为 YAML
```

**注释处理策略**：

```
格式转换时的注释处理：

YAML → JSON：
├── JSON 不支持注释，转换时注释会丢失
├── 在控制台输出警告提示用户
└── 数据内容完整保留

JSON → YAML：
├── 原文件无注释，生成的 YAML 也无注释
├── 可选择：生成带默认注释的模板格式
└── 数据内容完整保留

用户提示策略：
├── 转换前明确提示用户注释处理方式
├── 提供 --no-comment-warning 参数抑制警告
└── 转换后输出"注释已丢失"的摘要信息
```

**配置文件默认路径与搜索顺序**：

```
默认配置文件路径（初始化时未指定 --init-path）：
├── 当前目录 ./llm-probe.yaml（优先）
└── 用户配置目录
    ├── Linux/macOS: ~/.config/llm-probe/config.yaml
    └── Windows: %APPDATA%/llm-probe/config.yaml

配置文件搜索顺序（当未指定 -c 时）：
├── 1. 当前目录 ./llm-probe.yaml
├── 2. 当前目录 ./llm-probe.json
├── 3. 用户配置目录 ~/.config/llm-probe/config.yaml
└── 4. 用户配置目录 ~/.config/llm-probe/config.json
```

#### 3.2.3 服务商适配模块 (`provider/`)

**职责**：统一 LLM 服务调用接口，支持多服务商扩展

**核心抽象 - LLMBackend Trait**：

| 方法 | 功能 | 返回值 |
|------|------|--------|
| `chat_completion()` | 非流式对话调用 | ChatResponse (内容、耗时、Token统计) |
| `stream_chat()` | 流式对话调用 | 通过回调实时返回内容 |
| `list_models()` | 列出可用模型 | Vec<ModelInfo> |

**Provider 工厂逻辑**：

```
输入: provider 名称 + 配置信息
      │
      ▼
┌─────────────────────┐
│  Provider Factory   │
│  (匹配 provider)    │
└──────────┬──────────┘
           │
     ┌─────┴─────┬─────────┬─────────┐
     │           │         │         │
     ▼           ▼         ▼         ▼
 "openai"     "gemini"   "claude"   ...
     │           │         │         │
     ▼           ▼         ▼         ▼
OpenAI       Gemini     Claude     ...
Backend      Backend    Backend
     │           │         │         │
     └─────┬─────┴─────────┴─────────┘
           │
           ▼
   返回 Box<dyn LLMBackend>
```

**OpenAI Backend 实现逻辑**（基于 llm crate）：

```
初始化阶段：
├── 接收 api_key 和 base_url
├── 创建 llm crate 的 OpenAI 客户端实例
└── 配置 base_url（如自定义）

chat_completion 调用：
├── 记录开始时间
├── 构建 messages 列表（从 context 转换）
├── 调用 llm 客户端的 chat 方法
├── 等待完整响应
├── 计算耗时
└── 返回 ChatResponse

stream_chat 调用：
├── 构建 messages 列表
├── 调用 llm 客户端的 stream 方法
├── 循环读取 chunk
├── 每收到 chunk 立即调用回调函数
└── 直到流结束

list_models 调用：
├── 调用 llm 客户端的 models().list() 方法
└── 返回模型列表
```

**Gemini Backend 实现逻辑**：

```
与 OpenAI 类似，使用 llm crate 的 Gemini 模块
需注意：
├── API 格式差异（认证方式、请求结构）
├── base_url 处理方式不同
└── 错误码映射
```

> **注意**：本架构中关于 llm crate 的具体 API 调用方式需要根据实际文档和示例代码进行调整。后续开发阶段将根据提供的样例进行验证和修正。

**扩展新 Provider 步骤**：

1. 创建新文件 `provider/xxx.rs`
2. 实现 `LLMBackend` trait
3. 在 `factory.rs` 中添加匹配分支
4. 更新配置 schema 添加 provider 枚举值（可选）

#### 3.2.4 输出模块 (`output/`)

**职责**：格式化输出、流式显示、错误提示

**输出类型**：

| 类型 | 场景 | 格式 |
|------|------|------|
| **响应输出** | LLM调用结果 | 分隔线 + 模型名 + 耗时 + 内容 + Token统计(可选) |
| **模型列表** | `-l` 命令 | 标题 + 模型列表（带归属信息） |
| **错误输出** | 异常情况 | 红色高亮 + 用户友好提示 |
| **流式输出** | `--stream` 模式 | 实时打印，无缓冲 |

**响应输出格式**：

```
┌────────────────────────────────────────┐
│ 模型: gpt-4                            │  ← 绿色
│ 耗时: 1234 ms                          │  ← 黄色
├────────────────────────────────────────┤  ← 分隔线
│ 响应内容...                            │
└────────────────────────────────────────┘

Token 统计（可选，仅当 API 返回时显示）：
├── 显示格式: [Token: 输入 xx, 输出 xx]
└── 颜色：暗灰色（辅助信息）
```

**流式输出处理**：

```
┌─────────────────────────────────────────────────────────────┐
│                    流式输出流程                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 打印开始提示                                             │
│     ├── 模型名称（绿色）                                     │
│     └── "流式响应中..."                                      │
│                                                             │
│  2. 接收 chunk                                              │
│     ├── 立即打印（不缓冲）                                   │
│     ├── 立即 flush stdout                                   │
│     └── 循环直到流结束                                       │
│                                                             │
│  3. 流结束后                                                 │
│     ├── 打印换行                                            │
│     └── 可选：打印总耗时                                     │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**彩色输出规则**：

| 元素 | 颜色 | 说明 |
|------|------|------|
| 模型名称 | 绿色 | 重要信息高亮 |
| 耗时统计 | 黄色 | 性能指标 |
| 错误信息 | 红色 | 需要关注 |
| 分隔线 | 暗灰色 | 视觉分隔 |
| Token统计 | 暗灰色 | 辅助信息（可选显示） |

#### 3.2.5 错误处理模块 (`error/`)

**职责**：统一错误类型、用户友好提示

**错误类型分类**：

| 错误类型 | 场景 | 用户提示 |
|----------|------|----------|
| ConfigFormatError | 配置文件语法错误 | "配置文件格式错误，请检查 YAML/JSON 语法是否正确" |
| ConfigFileNotFound | 配置文件不存在 | "配置文件不存在: {path}" |
| InvalidApiKey | API Key 无效 | "API 密钥无效，请检查密钥是否正确或是否过期" |
| NetworkError | 网络连接失败 | "API 地址无法访问，请检查网络或地址是否正确" |
| ModelNotFound | 模型不存在 | "模型名称不存在，可通过 -l 参数查看当前服务商支持的模型列表" |
| UnsupportedProvider | 不支持的服务商 | "不支持的服务商: {provider}" |
| FileExists | 文件已存在 | 提示是否覆盖 |
| InputFileNotFound | 输入文件不存在 | "输入文件不存在，请检查路径是否正确" |
| LlmApiError | LLM API 错误 | 根据错误码映射友好提示 |

**HTTP 状态码映射规则**：

| HTTP 状态码 | 错误类型 | 用户提示 |
|-------------|----------|----------|
| 401 | InvalidApiKey | API 密钥无效，请检查密钥是否正确或是否过期 |
| 403 | InvalidApiKey | API 密钥无权限，请检查密钥权限范围 |
| 404 | ModelNotFound | 模型不存在，请检查模型名称或使用 -l 查看可用模型 |
| 429 | RateLimitError | 请求过于频繁，请稍后重试 |
| 500 | ServerError | 服务器内部错误，请稍后重试 |
| 502/503 | NetworkError | API 服务暂时不可用，请稍后重试 |
| 超时 | NetworkError | 请求超时，请检查网络连接或 API 地址 |

**错误处理流程**：

```
错误发生
    │
    ▼
转换为 LlmProbeError
    │
    ▼
调用 user_message() 获取用户友好提示
    │
    ▼
输出到 stderr（红色）
    │
    ▼
返回非零退出码
```

**敏感信息保护**：

- API Key 不出现在错误消息中
- API Key 不出现在日志中
- 使用 `SecretString` 包装敏感字段
- 仅在调用 API 时暴露

```
┌─────────────────────────────────────────────────────────────┐
│                    非流式调用流程                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 准备阶段                                                 │
│     ├── 根据 provider 选择 Backend                          │
```

#### 4.3.2 流式调用

```
┌─────────────────────────────────────────────────────────────┐
│                    流式调用流程                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 准备阶段                                                 │
│     ├── 根据 provider 选择 Backend                          │
│     ├── 将 context 转换为 messages 格式                      │
│     └── 打印开始提示（模型名称）                             │
│                                                             │
│  2. 建立流连接                                               │
│     ├── 调用 backend.stream_chat()                          │
│     └── 获取 stream 对象                                     │
│                                                             │
│  3. 流式读取循环                                             │
│     ┌─────────────────────────────────────┐                │
│     │  while stream.has_next()            │                │
│     │  ├── 读取下一个 chunk               │                │
│     │  ├── 立即打印到 stdout              │                │
│     │  ├── 立即 flush stdout              │                │
│     │  └── 捕获流错误                     │                │
│     └─────────────────────────────────────┘                │
│                                                             │
│  4. 流结束处理                                               │
│     ├── 打印换行                                            │
│     └── 可选：打印总耗时                                     │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

```
    │
```

**YAML 格式** (`llm-probe.yaml`)：

```yaml
# LLM Probe 配置文件
# 服务商：openai / gemini（可扩展）
provider: "openai"

# API 基础地址

base_url: "https://api.openai.com/v1"
```

## 7. 扩展性设计

### 7.1 新增服务商

**步骤**：

1. 在 `provider/` 目录创建新文件（如 `claude.rs`）
2. 实现 `LLMBackend` trait 的所有方法
3. 在 `factory.rs` 中添加 provider 匹配分支
4. 更新文档和配置示例

**扩展点设计**：

```
LLMBackend Trait 方法：
├── chat_completion() → 核心方法，必须实现
├── stream_chat()     → 核心方法，必须实现
└── list_models()     → 核心方法，必须实现
```

### 7.2 新增配置字段

**向后兼容策略**：

- 新字段使用 `Option<T>` 类型
- 使用 `#[serde(default)]` 注解
- 旧配置文件自动使用默认值
- 命令行参数逐步添加新选项

### 7.3 新增输出格式

**扩展点**：

- 通过 trait 定义输出格式化接口
